# 低空监管时空数据库运行验证测试指南

为了确保您能够顺利、正确地在本地 PostgreSQL 数据库中运行并验证本次大作业的成果，请严格按照以下步骤进行操作。

---

## 1. 软件环境准备

1. **PostgreSQL**：确保您已经安装了 PostgreSQL 数据库（建议 12 及以上版本）。
2. **PostGIS 空间扩展**：**这是必须的前置条件！** 空间数据库需要 PostGIS 支持。
   * **Windows 用户**：可以通过 PostgreSQL 自带的 `Application Stack Builder` 或 `PostGIS 独立安装包` 来安装。
   * 安装完成后，无需立刻配置，在后续步骤中我们会在数据库级别启用它。
3. **客户端工具**：建议使用图形化界面工具 **pgAdmin 4** 或 **Navicat**，当然命令行工具 **psql** 也是完全可以的。

---

## 2. 新建数据库并启用空间扩展

为了不污染您原来的默认库，我们强烈建议新建一个专门的数据库。

### 操作步骤（以 pgAdmin 为例）：
1. 在左侧的 Servers 树状目录中，右键点击 `Databases` -> `Create` -> `Database...`。
2. 数据库名称（Database）填入：`low_altitude_db`，点击保存。
3. 选中刚刚建好的 `low_altitude_db` 数据库，点击顶部的 **Query Tool** (查询工具) 图标，打开 SQL 编写面板。
4. 在查询面板中输入以下命令并运行：
   ```sql
   CREATE EXTENSION postgis;
   ```
   > [!IMPORTANT]
   > **验证这一步十分关键！** 如果运行成功，系统会提示 `CREATE EXTENSION`。
   > 如果提示 `could not open extension control file "postgis.control"`，说明您的电脑尚未安装 PostGIS 软件，请先下载并安装 PostGIS，然后再执行这一步。

---

## 3. 按顺序执行 SQL 脚本文件

请**严格按照以下 1 -> 2 -> 3 的顺序**加载并执行代码。您可以在 pgAdmin 的 Query Tool 中使用打开文件（Open File）图标加载这些 SQL 文件并运行。

### 第一步：执行建库建表脚本 (`schema.sql`)
1. **加载运行** `schema.sql`。
2. **预期现象**：由于脚本顶部包含了防重复建表的安全机制（`DROP TABLE IF EXISTS ...`），如果您是第一次运行，可能会看到类似 `table does not exist, skipping` 的提醒，**这是正常现象，请忽略它。**
3. **验证结果**：刷新左侧的 `Schemas` -> `public` -> `Tables`，您应该能看到刚刚生成的 **12 张表**（如 authority, disposal_record, drone, flight_mission 等）。

### 第二步：导入模拟测试数据 (`test_data.sql`)
1. **加载运行** `test_data.sql`。
2. **预期现象**：执行成功会提示 `INSERT 0 3`, `INSERT 0 4` 等信息。
3. **验证结果**：可以新建一个查询，输入 `SELECT count(*) FROM flight_mission;`，如果结果返回 `3`，说明飞行任务数据以及其它时空数据都已顺利导入。

### 第三步：运行核心时空验证查询 (`queries.sql`)
该脚本模拟了监管大屏的 4 个真实业务场景。为了更清晰地看到效果，建议您**分段高亮选中**（一次选中一个 SELECT 语句区块）来执行，观察每个场景的业务判定结果。

## 4. 验证查询的预期结果详解

以下是 `queries.sql` 脚本中四大查询对应的预期结果，用以向老师证明数据库的正确性：

### 🔍 场景一：飞行任务申报冲突检测
* **功能说明**：判断申请的路线边界（面）是否和管制区域（面）发生时间与空间上的冲突。
* **预期结果**：查询会抓出**1条冲突记录**（任务ID为2，执飞飞手：飞手乙，任务目的：大运周边快件派送测试）。
* **分析**：因为该任务申报的时间是今天 15:00-16:00，这正好与“龙岗体育馆大型演唱会临时禁飞区”（14:00-22:00生效）发生了时空交叠。

### 🔍 场景二：实时遥测轨迹侵入禁飞区警报（动态电子围栏）
* **功能说明**：识别高频上报的当前坐标（点）是否闯入了禁飞区（面）内部。
* **预期结果**：查询会精准抓取出**1条侵入记录**（时间为 12:46:00，飞手：张三，无人机：DJI Phantom 4 Pro）。
* **分析**：张三在此时的飞行高度为150m（在管控高度1000m以下），且二维空间坐标（点）完全落入了福田市民中心核心禁飞区内部（面），触发智能报警。

### 🔍 场景三：遥测高度超限违规检测
* **功能说明**：检测遥测高度是否超出了机型设计的物理最大升限。
* **预期结果**：查询会抓取出**1条超高记录**（所属单位：顺丰低空物流，无人机型号：丰翼丰舟-90，违规时间：13:10:00）。
* **分析**：该机型在数据库中设定的最大物理限高是 `300米`，而在 13:10 分传回的轨迹点高度高达 `350米`，越限差值 `50米` 被成功捕捉。

### 🔍 场景四：处置与执法闭环综合分析统计
* **功能说明**：从违规主体追踪到飞手、违规类型，再到监管局开具的罚单闭环聚合展示。
* **预期结果**：展示出一个管理报表。第一条记录显示“张三”被“民警现场查扣飞行设备”，结果为“罚款500元并暂扣无人机”；第二条记录显示“顺丰低空物流”被“警告并责令下降”，结果为“罚款企业5000元，记入征信”。

## 5. 常见问题排查 (FAQ)

* **Q：执行 queries.sql 提示 `relation "flight_mission" does not exist`？**
  * **A**：这必定是因为前面的步骤没有成功。请首先确保运行了 `CREATE EXTENSION postgis;`，然后再**完整无误地重新运行一遍 `schema.sql`**。只要 `schema.sql` 顺利建成了所有表，这个错误就会消失。
* **Q：执行 test_data.sql 提示外键约束冲突（foreign key constraint violation）？**
  * **A**：说明数据库内留存了脏数据。解决办法是：重新运行一次 `schema.sql`（它会自动清空旧表并重新建空表），然后紧接着只运行一次 `test_data.sql` 即可。
